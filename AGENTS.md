# 개발지침
모든 사항은 https://api.mateya.cloud/swagger 랑 https://docs.google.com/spreadsheets/d/1VNhXI8Ehj63yY0go3XjI4Dy2_SBwKuX-bQ9QXHlaL9U/edit?usp=sharing 참고해야한다.

백엔드 코드는 ../backend를 참고한다. backend db 변경은 backward-compatible하게 한다.
    
1. 개발 범위를 정한다.
    - 한 번에 처리할 작업 단위를 작게 나눈다.
    - 작업 범위가 불명확하거나 API/DB/인증/배포에 영향을 주는 경우 먼저 질문한다.
    - 사소한 불명확성은 기존 코드, 문서, 테스트를 확인한 뒤 가정을 명시하고 진행한다.

2. 최신 main 기준으로 작업 브랜치를 생성한다.
    - `git fetch origin`을 실행한다.
    - `origin/main` 또는 팀에서 지정한 기준 브랜치에서 새 브랜치를 생성한다.
    - 브랜치명에는 `codex`를 붙이지 않는다.
    - 브랜치명은 작업 의미가 드러나게 작성하고, 가능한 경우 `feat/`, `fix/`, `refactor/`, `chore/` prefix를 사용한다.

3. 개발을 진행한다.
    - 요청 범위를 벗어난 대규모 리팩터링, 의존성 추가, API 스펙 변경, DB 스키마 변경은 먼저 보고하고 확인받는다.
    - 기존 코드 스타일, 폴더 구조, 네이밍, 에러 처리 방식, 테스트 방식을 우선한다.
    - 보안 정보, 토큰, `.env`, 운영 서버 정보는 절대 커밋하지 않는다.

4. 작업 단위로 커밋한다.
    - 의미 있는 단위로 커밋한다.
    - 커밋 메시지에는 `codex`를 붙이지 않는다.
    - 커밋 메시지는 팀 규칙을 따르고, 가능한 경우 `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:` prefix를 사용한다.

5. 검증한다.
    - 관련 테스트, 린트, 타입체크, 빌드 중 가능한 검증을 실행한다.
    - 실행하지 못한 검증이 있다면 이유와 남은 리스크를 보고한다.
    - 프로젝트에 정의된 의존성 환경관리, 테스트코드 작성, 로깅 레벨 관련 Skill이 필요하면 해당 Skill을 사용한다.

6. 작업하면서 추가로 해야 할 내용이 있으면 `.agents/development-todo.md`에 추가한다.
    - 단, 현재 작업 범위와 무관한 아이디어성 항목은 임의로 추가하지 않는다.
    - 추가한 항목에는 이유와 우선순위를 간단히 적는다.

7. 작업이 끝났거나 중간 보고가 필요한 시점에 사용자에게 보고한다.
    - 변경 요약
    - 수정한 주요 파일
    - 실행한 검증 명령과 결과
    - 남은 리스크
    - 추가된 TODO

8. PR을 생성한다.
    - PR 제목에는 `codex`를 붙이지 않는다.
    - PR 제목은 작업 의미가 드러나게 작성하고, 가능한 경우 `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:` prefix를 사용한다.
    - PR 본문에는 변경 배경, 변경 내용, 영향 범위, 리뷰시 중점적으로 봐야할점을 포함한다.

9. squash merge도 알아서 진행한다.
    -  squash merge 메시지는 `PR제목 (#PR번호)` 형식을 따른다.

10.개발에 완료한 항목은 문서에서 삭제한다.
