# 모바일 앱 코드 작성 가이드

이 저장소는 `pubspec.yaml`, `android/`, `ios/` 구조를 가진 Flutter 프로젝트다. 따라서 별도 지시가 없으면 Android/iOS 앱 개발 문맥에서도 Flutter/Dart 코드를 기본 진입점으로 간주한다.

네이티브 Android/Kotlin 또는 iOS/Swift 코드는 아래 경우에만 수정한다.

- 플랫폼 SDK 직접 연동이 필요한 경우
- Flutter 플러그인 설정 또는 권한 설정이 필요한 경우
- 빌드 설정, 서명, 배포 메타데이터를 변경해야 하는 경우

UI, 상태 관리, 비즈니스 로직은 가능하면 `lib/` 안에서 해결하고, `android/`와 `ios/`에는 플랫폼별 필수 설정만 둔다.

## 기본 원칙

- 기존 프로젝트 구조와 패턴이 있으면 그것을 우선 따른다.
- 기존 패턴이 충분하지 않으면 Flutter 표준 구조를 따르되, 임시 분기보다 재사용 가능한 계층 분리를 우선한다.
- 한 화면에서 모든 로직을 처리하지 않는다. 화면, 상태, 도메인 로직, 외부 연동을 분리한다.
- Android/iOS 한쪽만 동작하는 구현을 넣을 때는 반대 플랫폼 영향과 fallback 동작을 같이 확인한다.
- 하드코딩된 문자열, 색상, 간격, API endpoint, 권한 키를 위젯 안에 직접 흩뿌리지 않는다.

## 코드 배치 원칙

- 공통 비즈니스 로직, 상태, 화면 코드는 `lib/` 하위에 둔다.
- 기능이 커지면 파일을 타입 기준보다 feature 기준으로 묶는다.
- 기본 권장 구조는 아래 순서를 따른다.
  - `lib/features/<feature>/presentation/`
  - `lib/features/<feature>/application/` 또는 `state/`
  - `lib/features/<feature>/domain/`
  - `lib/features/<feature>/data/`
- 여러 화면에서 함께 쓰는 컴포넌트는 `lib/shared/` 또는 `lib/core/` 성격의 디렉토리로 분리한다.
- 플랫폼별 설정 파일은 해당 디렉토리에 유지한다.
  - Android: `android/app/src/...`, `AndroidManifest.xml`, Gradle 설정
  - iOS: `ios/Runner/...`, `Info.plist`, Xcode project 설정

## 리팩토링 작업 기준

- 500줄 이상 파일은 우선적으로 분리 후보로 본다.
- 단순히 private 메서드만 떼지 말고, 책임 단위로 파일을 나눈다.
- 화면 파일은 가능한 한 진입점과 coordinator 역할만 남기고, 실제 섹션 UI는 `presentation/widgets/`로 분리한다.
- `application/` 계층은 상태 전이와 사용자 액션 orchestration을 중심에 두고, 검증 로직, 이미지 처리, 검색 보조 로직처럼 독립 가능한 책임은 helper 또는 별도 파일로 뺀다.
- `data/` 계층은 repository 정의, API 구현, mock 구현, fixture, mapper, 업로드/파싱 보조 책임을 한 파일에 몰아두지 않는다.
- 기존 public API, constructor signature, 라우팅 진입점, 상태 흐름은 특별한 이유가 없으면 유지한다.
- 파일을 나눈 뒤 import 경로만 바꾸고 동작을 바꾸지 않는 것이 우선이며, 기능 변경은 별도 작업으로 다룬다.
- 기존 테스트가 있는 계층부터 먼저 건드리고, 한 작업 단위가 끝날 때마다 `dart format`, `flutter analyze`, 관련 `flutter test`를 바로 실행한다.
- 현재 범위와 직접 관련 있는 후속 작업만 `.agents/development-todo.md`에 추가하고, 이유와 우선순위를 함께 남긴다.

## Flutter UI 작성 원칙

- 가능한 한 `StatelessWidget`을 우선 사용하고, 로컬 UI 상태가 실제로 필요할 때만 `StatefulWidget`을 사용한다.
- `build()` 메서드 안에 네트워크 호출, 저장소 접근, 무거운 계산을 넣지 않는다.
- 위젯 하나가 너무 커지면 역할별 하위 위젯으로 분리한다.
- 간격과 여백은 일관된 spacing 규칙을 사용하고, 매직 넘버를 반복하지 않는다.
- 테마를 우선 사용하고, 개별 위젯에 색상과 텍스트 스타일을 반복 지정하지 않는다.
- 접근성을 위해 버튼 의미, 터치 영역, 텍스트 대비, 스크린리더 친화성을 고려한다.

## 상태 관리 원칙

- 상태 변경 로직을 위젯 트리 내부에 과도하게 섞지 않는다.
- 화면 상태와 도메인 상태를 구분한다.
  - 화면 상태 예: 탭 선택, 로딩 노출 여부
  - 도메인 상태 예: 로그인 세션, 사용자 정보, 주문 상태
- 비동기 흐름은 이벤트, 상태, 실패 케이스가 드러나도록 작성한다.
- 전역 상태는 꼭 필요한 범위에만 공유하고, 단순 화면 로컬 상태까지 전역으로 올리지 않는다.
- 이미 도입된 상태 관리 도구가 있으면 그 도구를 유지하고, 근거 없이 다른 패턴을 섞지 않는다.

## 데이터 및 네트워크 계층

- API 호출 코드를 위젯에서 직접 작성하지 않는다.
- repository, service, datasource 등 외부 연동 계층을 분리한다.
- JSON 파싱, DTO 변환, 도메인 모델 변환 책임을 한 곳에 몰아넣지 말고 경계를 분리한다.
- 실패는 무시하지 말고, 호출자에게 에러 상태나 예외를 명확히 전달한다.
- 토큰, 사용자 식별자, 디바이스 식별자, 권한 상태 같은 민감한 값은 로그에 그대로 남기지 않는다.

## Android/iOS 플랫폼 연동 원칙

- Flutter만으로 해결 가능한 문제를 불필요하게 네이티브 코드로 내리지 않는다.
- 네이티브 수정이 필요하면 Android와 iOS 양쪽 설정 위치를 함께 확인한다.
- 권한 추가 시 선언만 하지 말고 실제 요청 흐름과 거부 시 UX까지 같이 고려한다.
- 딥링크, URL scheme, 푸시, 백그라운드 처리처럼 플랫폼 의존성이 큰 기능은 플랫폼별 차이를 코드와 설정에 명확히 드러낸다.
- 빌드 설정 변경 시 앱 식별자, 서명, 최소 지원 버전, 권한 설명 문구에 미치는 영향을 먼저 확인한다.

## 하위 호환성

- 이미 배포된 앱의 package name, bundle identifier, 딥링크 경로, 권한 키 이름을 함부로 바꾸지 않는다.
- 로컬 저장소 키, 캐시 키, 직렬화 필드명은 기존 사용자 데이터와 충돌하지 않도록 유지한다.
- API 응답 모델 변경 시 구버전 앱과 서버 간 호환성 영향을 먼저 확인한다.
- 저장 데이터 구조 변경이 필요하면 마이그레이션 또는 fallback 처리 기준을 먼저 설계한다.

## 네이밍 규칙

- 클래스명은 역할이 드러나도록 작성한다.
  - 화면: `HomePage`, `LoginScreen`
  - 상태 객체: `LoginState`
  - 컨트롤러/뷰모델: `ProfileController`, `FeedViewModel`
  - 저장소: `AuthRepository`
- bool 값은 `is`, `has`, `can`, `should` 접두어를 우선 사용한다.
- 날짜/시간 값은 의미가 분명하면 `_at` 계열 이름을 우선 사용한다.
  - 예: `createdAt`, `updatedAt`, `expiresAt`
- 축약어는 팀에서 이미 굳어진 경우가 아니면 남발하지 않는다.

## 에러 처리 원칙

- 실패를 `print`로만 남기고 끝내지 않는다.
- 사용자에게 보여줄 오류와 내부 디버깅 오류를 구분한다.
- 네트워크 실패, 인증 만료, 권한 거부, 빈 데이터, 파싱 실패를 같은 에러로 뭉개지 않는다.
- 재시도 가능한 오류와 즉시 사용자 조치가 필요한 오류를 구분한다.

## 비동기 처리 원칙

- `async` 호출 결과를 무시하지 않는다.
- 로딩 상태, 성공 상태, 실패 상태를 명시적으로 다룬다.
- 화면 dispose 이후 상태 갱신이 일어나지 않도록 생명주기를 고려한다.
- debounce, throttle, cancelation이 필요한 입력 흐름은 임시 타이머 분기보다 명시적 제어 구조를 사용한다.

## 리소스 및 설정 관리

- 이미지, 폰트, 기타 asset은 `pubspec.yaml`에 명시적으로 등록한다.
- 환경별 상수, API base URL, feature flag는 코드에 직접 박아 넣지 않는다.
- Android/iOS 설정값은 서로 다른 위치에 중복될 수 있으므로 한쪽만 수정하고 끝내지 않는다.

## 코드 포맷팅 및 검증

포맷팅은 `dart format`을 기본으로 사용한다.

```bash
dart format lib test
```

Flutter 정적 검사는 `flutter analyze`를 우선 사용한다.

```bash
flutter analyze
```

변경 범위에 테스트가 있다면 `flutter test`로 가장 가까운 테스트부터 확인한다.

```bash
flutter test
```

## Agent 작업 원칙

- 먼저 현재 저장소가 Flutter인지, React Native인지, 네이티브 Android/iOS인지 확인하고 명령과 규칙을 적용한다.
- 이 저장소에서는 별도 지시가 없으면 Flutter 프로젝트로 간주한다.
- 새 아키텍처를 도입할 때는 기존 코드가 없는 상태인지, 기존 패턴이 약한 상태인지 먼저 확인하고 최소한의 일관된 규칙만 추가한다.
- 불확실한 플랫폼 정책, 배포 설정, 서명, 프로비저닝 관련 정보는 추측하지 말고 사용자에게 질문한다.
