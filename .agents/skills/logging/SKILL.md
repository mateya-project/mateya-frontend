# Flutter 로깅 가이드

이 스킬은 Flutter 기반의 iOS/Android 앱을 개발, 출시, 운영하는 문맥에서 실무적으로 유지 가능한 로깅 체계를 설계하거나 구현할 때 사용한다. 목표는 "콘솔에 문자열을 많이 찍는 것"이 아니라, 디버깅 가능성, 운영 관측성, 개인정보 보호, 비용 통제를 동시에 만족하는 것이다.

## 먼저 확인할 것

로깅 작업을 시작하기 전에 아래를 먼저 확인한다.

- `pubspec.yaml`: 이미 사용 중인 로깅/모니터링 패키지가 있는지 확인
- `lib/main.dart`: 앱 시작점, 전역 에러 핸들링 위치 확인
- `ios/`, `android/`: 네이티브 초기화 코드와 크래시 리포팅 연동 상태 확인
- 네트워크 레이어: `Dio`, `http`, GraphQL 클라이언트 등 인터셉터 삽입 지점 확인
- DI / 상태관리: `Provider`, `Riverpod`, `Bloc`, `GetIt` 등 로거 주입 방식 확인
- 운영 도구: Firebase, Sentry, Datadog 등 이미 조직 표준이 있는지 확인
- 배포 방식: iOS TestFlight / App Store, Android internal / closed / production track 구분 확인

기존 스택이 있으면 그 규칙을 우선 따른다. 이미 `logger`를 쓰는 프로젝트에 갑자기 `talker`를 추가하거나, 이미 Sentry를 쓰는 프로젝트에 별 이유 없이 Crashlytics를 병행 도입하지 않는다.

## 질문이 필요한 경우

아래 정보가 불명확하면 추정하지 말고 먼저 질문한다.

- 조직 표준 원격 에러 수집 도구가 있는지
- 개인정보, 토큰, 전화번호, 이메일 등 민감정보 로그 허용 범위
- release 빌드에서 어느 수준까지 로그를 남겨야 하는지
- 서버 로그와 사용자 세션을 연결할 trace / request id 정책이 있는지
- iOS / Android 네이티브 크래시까지 하나의 대시보드로 볼지 여부

## 핵심 원칙

- `print()` 남발을 로깅 전략으로 취급하지 않는다.
- 로컬 디버그 로그, 운영 에러 리포팅, 제품 분석 이벤트를 서로 다른 목적으로 분리한다.
- 로그는 "많이"보다 "의미 있게" 남긴다.
- 민감정보는 기본적으로 기록하지 않고, 꼭 필요하면 마스킹 규칙을 둔다.
- release 환경에서는 verbose 로그를 줄이고, 에러와 핵심 진단 정보만 남긴다.
- UI `build()` 호출, 스크롤, 애니메이션 프레임처럼 노이즈가 큰 지점은 로그를 남기지 않는다.

## 권장 구조

실무에서는 로깅을 아래 3층으로 나누는 것이 안전하다.

1. 앱 내부 진단 로그
2. 예외 / 크래시 수집
3. 분석 이벤트

이 셋을 하나의 도구로 억지로 해결하려 하지 않는다.

### 1. 앱 내부 진단 로그

기본 방향:

- 패키지 위에 `AppLogger` 같은 얇은 추상화 레이어를 둔다.
- 화면, 유스케이스, 리포지토리, 네트워크 계층에서 같은 인터페이스를 사용하게 만든다.
- 문자열만 찍지 말고 `message`, `error`, `stackTrace`, `context`를 함께 전달할 수 있게 한다.

패키지 선택 기준:

- 단순하고 가벼운 로컬 로그면 `logger`
- 로그 뷰어, 네비게이션 추적, 개발 중 가시성이 중요하면 `talker`

둘 중 하나만 선택한다. 특별한 이유가 없다면 보통은 `logger`로도 충분하다.

### 2. 예외 / 크래시 수집

선택 기준:

- 이미 Firebase를 쓰고 있으면 `Firebase Crashlytics`
- 앱/백엔드/성능 추적을 함께 보고 싶으면 `sentry_flutter`

둘 다 좋은 도구지만 운영 표준 없이 병행 도입하면 비용과 복잡도만 증가한다.

iOS/Android 운영 관점에서는 Flutter Dart 예외만 모아서는 부족하다. 아래를 함께 본다.

- Flutter 프레임워크 예외
- Dart 비동기 예외
- iOS 네이티브 크래시
- Android 네이티브 크래시
- Android ANR, OOM 징후, 비정상 종료 패턴

### 3. 분석 이벤트

Amplitude, Firebase Analytics, Mixpanel 같은 제품 분석 이벤트는 로깅과 분리한다. "버튼 클릭", "구매 완료" 같은 데이터는 분석 시스템으로 보내고, 개발 로그로 대체하지 않는다.

## 로그 레벨 기준

프로젝트 전반에 아래 기준을 일관되게 적용한다.

- `debug`: 개발 중 상세 흐름 확인용
- `info`: 정상적인 주요 상태 변화
- `warning`: 복구 가능하지만 비정상적인 상태
- `error`: 기능 실패 또는 예외 발생
- `fatal`: 앱 중단 가능성이 큰 치명적 오류

release 환경에서는 보통 `info` 이상만 남기고, `debug`는 끄거나 샘플링한다.

## 어디에 로그를 남길 것인가

남겨야 하는 곳:

- 앱 시작과 의존성 초기화
- 로그인, 로그아웃, 토큰 갱신 실패
- API 실패, 재시도, 타임아웃, 역직렬화 실패
- 권한 요청 거절, 로컬 저장소 읽기/쓰기 실패
- 주요 도메인 상태 전이
- 백그라운드 작업 실패
- 사용자 제보를 재현하는 데 필요한 핵심 분기

남기지 말아야 하는 곳:

- `build()`가 자주 호출되는 위젯
- 매 프레임/스크롤 이벤트
- 비밀번호, access token, refresh token, 주민번호, 카드번호 등 민감정보
- 성공/실패 판단에 의미 없는 반복 메시지

## 민감정보 처리

다음 값은 기본적으로 로그에 남기지 않는다.

- 액세스 토큰, 리프레시 토큰, 세션 쿠키
- 이메일, 전화번호, 주소, 계좌번호
- 주민등록번호, 생년월일, 카드번호
- 원문 API payload 전체

필요하면 helper를 두고 일부만 남긴다.

- 이메일: `ab***@domain.com`
- 전화번호: `010-****-1234`
- 토큰: 앞 6자리만 남기고 나머지 마스킹

마스킹 규칙은 각 파일에서 즉흥적으로 처리하지 말고 공용 함수로 통일한다.

## Flutter 구현 지점

실무에서는 아래 지점을 우선 잡는다.

### 앱 시작점

`main()`에서 로거, 원격 에러 수집기, 전역 에러 핸들러를 초기화한다.

- `FlutterError.onError`
- `PlatformDispatcher.instance.onError`
- `runZonedGuarded`

이 세 지점을 통해 Flutter 프레임워크 에러, 플랫폼 비동기 에러, 존 바깥 예외를 최대한 한 곳으로 모은다.

### 네이티브 경계

Flutter 앱이라도 출시 후에는 iOS/Android 네이티브 영역 문제가 반드시 발생한다. 예를 들면:

- 플러그인 초기화 실패
- 푸시 권한 및 토큰 등록 실패
- 웹뷰, 카메라, 파일, 위치 권한 처리 문제
- 백그라운드 전환 중 플랫폼 예외

따라서 Dart 로거만 설계하지 말고, 선택한 원격 에러 수집 도구가 iOS/Android 네이티브 크래시와 연동되는지 확인한다.

### 네트워크 레이어

`Dio`를 쓰면 interceptor에서 아래를 기록한다.

- 요청 메서드와 path
- request id / trace id
- 응답 status code
- latency
- 실패 원인

본문 전체를 그대로 남기기보다 필요한 필드만 선별하고, 민감정보는 마스킹한다.

### 상태관리 레이어

`Bloc`, `Riverpod`, `ChangeNotifier`, `Cubit` 등 상태 전이가 중요한 프로젝트라면, 디버그 환경에서만 주요 상태 변화를 기록한다. 단, 모든 상태를 기계적으로 찍지 말고 재현에 필요한 이벤트만 남긴다.

### 앱 라이프사이클

운영 중 장애 분석을 위해 아래 이벤트는 필요 시 기록한다.

- foreground -> background 전환
- cold start / warm start
- 로그인 세션 만료 후 재진입
- 푸시 딥링크 진입
- 백그라운드 작업 시작/실패

단, 사용자의 모든 행동을 로깅하라는 뜻은 아니다. 장애 재현과 운영 판단에 필요한 이벤트만 남긴다.

## 권장 구현 방식

실무적으로는 아래 순서를 권장한다.

1. `AppLogger` 인터페이스 정의
2. `logger` 또는 `talker` 기반 구현체 추가
3. `main()`에서 전역 에러 핸들러 연결
4. 네트워크 인터셉터 연결
5. 마스킹 유틸 추가
6. release / debug 환경별 로그 레벨 분리
7. 필요 시 Sentry 또는 Crashlytics 연결

로깅 패키지를 화면 코드에서 직접 호출하기보다, 공용 로거를 DI로 주입하는 편이 테스트와 교체가 쉽다.

## 출시/운영 기준

모바일 앱은 배포 후에야 드러나는 문제가 많다. 따라서 로깅은 개발 단계에서 끝나지 않고 출시 파이프라인과 연결되어야 한다.

### 환경 분리

- `dev`, `staging`, `production` 환경을 구분한다.
- DSN, API key, 로그 레벨, 샘플링 비율을 환경별로 분리한다.
- staging 이벤트와 production 장애가 같은 대시보드에서 섞이지 않게 한다.

### 심볼 업로드

릴리즈 빌드의 스택트레이스를 읽으려면 심볼 파일 관리가 필요하다.

- iOS: dSYM 업로드 확인
- Android: ProGuard / R8 mapping 파일 업로드 확인

이 설정이 빠지면 운영 장애가 발생해도 스택트레이스가 난독화되어 원인 분석이 크게 늦어진다.

### 릴리즈 전 확인 항목

- release 빌드에서 `debug` 로그가 과도하게 남지 않는지
- 민감정보가 원격 리포트에 포함되지 않는지
- 앱 시작 시 전역 에러 핸들러가 실제로 연결되는지
- 강제 예외 테스트 시 원격 도구에 이벤트가 정상 수집되는지
- iOS/Android 양쪽 모두에서 심볼화가 되는지

### 운영 중 봐야 할 지표

- 크래시 프리 유저 비율
- 신규 버전 배포 직후 에러 급증 여부
- 특정 OS 버전 / 기기 모델 집중 장애
- 네트워크 타임아웃 증가
- 로그인, 결제, 푸시 진입 등 핵심 플로우 실패율

장애 대응 시에는 "에러 건수"만 보지 말고, 사용자 수 기준 영향도와 특정 앱 버전 편중 여부를 같이 본다.

## 금지 패턴

- `print()`를 앱 전역 표준처럼 사용하는 것
- 패키지별 API를 여기저기서 직접 호출하는 것
- 같은 에러를 여러 계층에서 중복 리포팅하는 것
- release에서도 요청/응답 원문 전체를 남기는 것
- 예외를 삼킨 뒤 아무 로그도 남기지 않는 것
- 임시 디버그 로그를 기능 완료 후 정리하지 않는 것
- iOS/Android 릴리즈 심볼 업로드를 빠뜨린 채 운영에 들어가는 것
- production 장애와 staging 테스트 이벤트를 같은 프로젝트에 섞는 것

## 패키지 추가 예시

로컬 진단 로그:

```bash
flutter pub add logger
```

Firebase 기반 크래시 수집:

```bash
flutter pub add firebase_crashlytics
```

Sentry 기반 크래시 수집:

```bash
flutter pub add sentry_flutter
```

단, 어떤 원격 도구를 추가할지는 조직 표준과 기존 인프라를 먼저 확인한 뒤 결정한다.

## 테스트 원칙

- 로거 자체보다 "어떤 상황에서 어떤 로그/리포트가 발생해야 하는지"를 테스트한다.
- 실패 재현에 중요한 분기에서는 mock logger를 주입해 호출 여부를 검증한다.
- 원격 에러 수집 도구는 테스트에서 실제 전송하지 않고 adapter를 mock 처리한다.

## Agent 작업 원칙

- 프로젝트에 이미 로깅 레이어가 있으면 확장 방향으로 작업하고 새 표준을 임의로 만들지 않는다.
- 운영 도구 선택이 불명확하면 질문부터 한다.
- 단순 콘솔 출력 추가로 끝내지 말고, 전역 에러 처리와 민감정보 정책까지 함께 본다.
- "지금 당장 보이게만 하는" 임시 로그보다, 제거와 유지보수가 쉬운 구조를 우선한다.
- iOS/Android 출시 운영 문맥이면 Flutter 코드만 보지 말고 네이티브 연동, 심볼 파일, 배포 환경 분리까지 함께 점검한다.
