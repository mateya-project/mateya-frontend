# 백엔드 전달사항

현재 Swagger: https://api.mateya.cloud/swagger  
기능정의서: https://docs.google.com/spreadsheets/d/1VNhXI8Ehj63yY0go3XjI4Dy2_SBwKuX-bQ9QXHlaL9U/edit?gid=0#gid=0

## 먼저 공유할 전제
- 아래 항목은 `2026-06-18` 기준 프론트 구현에 맞춰 정리한 백엔드 필요사항이다.
- 프론트는 우선 화면/플로우를 완성했고, 일부는 현재 mock/local state로 동작한다.
- 가능하면 기존 응답 shape를 재사용하고, 꼭 새 API가 필요할 때만 추가해주면 된다.
- 인증 필요한 API는 모두 로그인 세션 기준 `me` 처리가 가능하면 그쪽이 프론트 연결에 유리하다.
- 특히 번역/기존회원 판별처럼 별도 액션 API 없이 기존 조회/인증 응답만 보강해도 되는 항목은 그 방식이 프론트 연결 비용이 가장 낮다.

## 우선순위

### P1
1. 신고하기
2. 유저 차단 / 차단해제 / 차단목록 조회
3. 즐겨찾기 목록 조회
4. 모임 참여신청 승인/제거/대기목록 조회
5. 전화번호 인증 후 기존 가입회원 여부 + 기존 동네정보 반환

### P2
6. 채팅 메시지 번역/원문 보기 응답 명확화
7. 리뷰 번역/원문 보기 응답 명확화
8. 개인정보 수집·이용 동의 내역 조회
9. 활동 카테고리 메타데이터 조회
11. 활동 상세의 사용자 상태 응답 보강
12. 1:1 채팅방 표시용 제목/상대 정보 보강

## 1. 신고하기

### 필요한 이유
- 프론트에는 신고 시트 UI가 이미 있음.
- 현재는 제출 결과를 저장하지 못함.

### 요청사항
- 신고 본문 최대 `500자`
- 이미지 최대 `5장`
- 신고 대상 타입 구분 필요
  - 예: `USER`, `ACTIVITY`, `CHAT_MESSAGE`, `REVIEW`
- 신고자 정보는 서버가 인증 세션으로 식별
- 신고 접수 시간은 서버 저장값 기준 사용
- 접수 시 Discord 알림 전송

### 권장 API
- `POST /api/v1/reports`

### 요청 바디 예시
```json
{
  "targetType": "USER",
  "targetId": "123",
  "reason": "부적절한 메시지를 반복적으로 보냈습니다.",
  "imageUrls": [
    "https://..."
  ]
}
```

### 응답 예시
```json
{
  "id": 1,
  "targetType": "USER",
  "targetId": "123",
  "status": "RECEIVED",
  "createdAt": "2026-06-18T12:34:56Z"
}
```

### 참고
- 업로드는 기존 presigned 업로드 플로우 재사용 가능.
- 프론트는 URL 배열만 있으면 됨.

## 2. 유저 차단하기

### 필요한 이유
- 프론트에 타인 프로필 하단 `유저차단하기`와 설정의 `차단 유저 목록` 화면이 추가됨.

### 요청사항
- 차단 시 기존 친구 관계가 있으면 자동 삭제
- 차단한 상대와 다시 친구 맺기 불가
- 차단한 상대 프로필 상세 조회 불가
- 차단은 자기 자신에게 불가
- 중복 차단 요청은 멱등 처리 권장

### 권장 API
- `POST /api/v1/users/{targetUserId}/block`

### 응답 예시
```json
{
  "targetUserId": 123,
  "blocked": true,
  "friendRemoved": true,
  "blockedAt": "2026-06-18T12:34:56Z"
}
```

### 에러 정책 권장
- 자기 자신 차단 시 `400`
- 이미 차단된 경우 `200` 또는 `409` 중 하나로 통일

### 기존 조회 API에도 필요한 동작
- 최소한 `GET /api/v1/users/{targetUserId}` 호출 시, 내가 차단한 사용자 상세 프로필은 조회되지 않도록 처리 필요
- 응답은 `403` 또는 `404` 중 하나로 통일하고, 프론트 분기용 에러 코드도 함께 주는 편이 좋음
  - 예: `blocked-user`

## 3. 유저 차단해제하기

### 요청사항
- 차단 해제 후 관계는 `친구아닌 일반 유저` 상태로 복귀
- 자동으로 친구 복구되면 안 됨

### 권장 API
- `DELETE /api/v1/users/{targetUserId}/block`

### 응답 예시
```json
{
  "targetUserId": 123,
  "blocked": false
}
```

## 4. 차단 유저 목록 보기

### 필요한 이유
- 설정 > 차단 유저 목록 화면 구현 완료
- 각 유저별 이름, 프로필 사진, 거주지역 표시 필요

### 권장 API
- `GET /api/v1/users/me/blocked-users`

### 정렬
- 최신 차단순

### 응답 예시
```json
{
  "items": [
    {
      "userId": 123,
      "displayName": "Lee MinJi 이민지",
      "profileImageUrl": "https://...",
      "activityRegionName": "Living in Seoul · Shindang dong",
      "blockedAt": "2026-06-18T12:34:56Z"
    }
  ]
}
```

### 프론트가 실제로 쓰는 필드
- `userId`
- `displayName`
- `profileImageUrl`
- `activityRegionName` 또는 동등한 거주지역 문자열

### 있으면 좋은 필드
- `blockedAt`
  - 최신 차단순 정렬 확인용
  - 나중에 차단일시 노출이 필요해도 그대로 재사용 가능

## 5. 즐겨찾기 목록 보기

### 필요한 이유
- 홈/둘러보기 우하단 FAB로 `즐겨찾기 목록` 화면 진입 구현 완료
- 현재는 로컬 목록만 노출 중

### 권장 API
- `GET /api/v1/users/me/favorite-activities`

### 정렬
- 즐겨찾기 등록 최신순

### 응답 shape
- 가능하면 기존 `GET /api/v1/activities` 카드 응답과 최대한 동일하게 맞춰주면 좋음
- 첫 버전은 전체 목록 반환으로도 충분함
- 추후 페이지네이션이 필요하면 `GET /api/v1/activities`와 같은 `items/page/size/hasNext/nextPage` shape 재사용 권장

### 프론트가 필요한 필드
- `id`
- `title`
- `category`
- `placeName` 또는 `placeAddress`
- `startAt`
- `endAt`
- `priceType`
- `priceAmount`
- `participantCount`
- `capacity`
- `reviewRating`
- `imageUrl`
- 가능하면 `favoritedAt`

## 6. 모임 참여신청 목록 조회

### 필요한 이유
- 모임 상세에서 `참여내역` 버튼을 누르면 참여신청 관리 화면으로 이동하도록 프론트 구현 완료
- 현재는 mock 데이터만 사용 중
- 일반 상세 조회용 `participantPreviews`와 별개로, 호스트 전용 관리 화면에 필요한 승인/대기 목록이 필요함

### 권장 API
- `GET /api/v1/activities/{activityId}/participants/manage`

### 응답 예시
```json
{
  "participantCount": 18,
  "capacity": 25,
  "approvedParticipants": [
    {
      "userId": 1,
      "displayName": "Lee MinJi 이민지",
      "profileImageUrl": "https://...",
      "activityRegionName": "Living in Seoul · Shindang dong"
    }
  ],
  "pendingParticipants": [
    {
      "userId": 2,
      "displayName": "Park Jun 박준",
      "profileImageUrl": "https://...",
      "activityRegionName": "Living in Seoul · Yeouido dong",
      "requestedAt": "2026-06-18T12:34:56Z"
    }
  ]
}
```

### 프론트가 필요한 필드
- 상단 프로그레스 계산용 `participantCount`, `capacity`
- 승인 목록 / 대기 목록 각각
  - `userId`
  - `displayName`
  - `profileImageUrl`
  - `activityRegionName`

### 권한/정렬
- 호스트만 조회 가능
- 비호스트 접근 시 `403` 권장
- `pendingParticipants`는 최신 신청순이면 관리 UX에 가장 잘 맞음

## 7. 모임 참여신청 승인

### 필요한 이유
- 대기 목록의 `+` 버튼을 누르면 승인 목록으로 이동하는 UX 구현 완료

### 권장 API
- `POST /api/v1/activities/{activityId}/participants/{userId}/approve`

### 응답
- 승인 후 최신 참가자 현황 반환 또는 `204`
- 프론트 연결 편의상 최신 현황 재조회 없이 쓸 수 있게 아래 정도는 반환 권장
```json
{
  "participantCount": 19,
  "capacity": 25,
  "approved": true
}
```

### 비즈니스 룰
- 정원 초과 시 승인 불가
- 이미 승인된 유저면 멱등 처리 권장
- 호스트만 승인 가능

### 에러 코드 예시
- `capacity-exceeded`
- `not-host`
- `participant-not-found`

## 8. 모임 참여자 제거 / 신청 취소 처리

### 필요한 이유
- 승인 목록에서 체크→삭제, 슬라이드 삭제 UX 구현 완료
- 대기 목록도 슬라이드해서 취소/삭제 UX 구현 완료

### 권장 API 분리
- 승인된 참여자 제거
  - `DELETE /api/v1/activities/{activityId}/participants/{userId}`
- 대기 신청 취소/반려
  - `DELETE /api/v1/activities/{activityId}/participant-requests/{userId}`

### 응답
- `204` 또는 최신 카운트 반환

### 비즈니스 룰
- 호스트만 가능
- 이미 없는 유저에 대한 삭제는 멱등 처리 권장

### 구분 이유
- 프론트 화면에서 승인된 참여자 삭제와 대기 신청 반려가 UX상 분리되어 있어서 API도 분리되어 있으면 연결이 단순함

## 9. 채팅 메시지 번역하기 / 원문보기

### 현재 프론트 기대사항
- 프론트는 별도 번역 API 호출 없이 메시지 응답 안에 원문/번역문이 같이 있으면 토글 가능
- 현재 파서 기준:
  - `text`: 현재 표시용 번역문 또는 기본 본문
  - `originalText`: 원문
- `text`와 `originalText`가 모두 있고 서로 다를 때만 `번역 보기 / 원문 보기` 토글이 활성화됨

### 확인 또는 보강 필요
- 기존 채팅 메시지 조회 응답에서 아래 두 필드를 안정적으로 내려주는지 확인 필요
  - `text`
  - `originalText`

### 응답 규칙 제안
- 번역본이 없으면 `text`만 내려줘도 됨
- 번역본이 있고 원문과 다를 때만 `originalText`를 같이 내려주면 프론트 토글이 자연스럽게 동작함

### 새 API가 꼭 필요하다면
- `POST /api/v1/chats/messages/{messageId}/translate`
- 하지만 프론트는 현재 `메시지 조회 시 함께 반환` 방식이 더 붙이기 쉬움

## 10. 리뷰 번역하기 / 원문보기

### 현재 프론트 기대사항
- 리뷰도 별도 번역 API 호출 없이 응답 안에 두 필드가 있으면 토글 가능
- 현재 파서 기준:
  - `body`: 현재 표시용 번역문 또는 기본 본문
  - `originalBody`: 원문
- `body`와 `originalBody`가 모두 있고 서로 다를 때만 토글이 활성화됨

### 확인 또는 보강 필요
- 리뷰 조회 응답에서 아래 두 필드 보장 필요
  - `body`
  - `originalBody`

### 응답 규칙 제안
- 번역본이 없으면 `body`만 내려줘도 됨
- 번역본이 있고 원문과 다를 때만 `originalBody`를 같이 내려주면 프론트 토글이 자연스럽게 동작함

### 관련 기존 API
- `GET /api/v1/activities/{activityId}/reviews`

## 11. 전화번호 인증 후 기존 가입회원 여부 + 기존 동네정보 반환

### 현재 프론트 플로우
- `POST /api/v1/auth/sms/verify`로 `verificationToken` 발급
- 이후 `POST /api/v1/auth/login` 성공 여부로 기존 회원인지 판단
- 기존 회원이면 온보딩 완료로 바로 가지 않고, `기존 동네정보를 보여준 뒤 동네 재인증` 화면으로 이동하도록 프론트 변경 완료

### 현재 프론트가 이미 기대하는 호환 규칙
- 아직 `verify` 응답에 `existingUser`가 없다면, 현재 프론트는 `POST /api/v1/auth/login`이 `404 not-found`일 때만 신규 회원으로 판단함
- 즉, 이 구간의 에러 정책을 바꿀 경우 프론트 분기 로직도 같이 맞춰야 함

### 현재 문서에서 보강이 필요한 부분
- 단순히 `재가입인지 여부`만 반환하면 부족함
- 프론트는 아래 정보가 같이 필요함
  - 기존 회원 여부
  - 기존 동네명
  - 가능하면 기존 위경도

### 권장 방향 A
- `POST /api/v1/auth/login` 응답의 `user` 안에 기존 값 보장
  - `activityRegionName`
  - `activityLatitude`
  - `activityLongitude`

### 권장 방향 B
- 또는 `POST /api/v1/auth/sms/verify` 응답에 아래 필드 추가
```json
{
  "verificationToken": "...",
  "expiresAt": "...",
  "existingUser": true,
  "previousActivityRegionName": "우만동",
  "previousActivityLatitude": 37.123,
  "previousActivityLongitude": 127.123
}
```

### 프론트 관점 추천
- 현재 코드 흐름상 `login` 응답의 `user.activityRegionName`만 안정적으로 내려줘도 우선 연결 가능
- 다만 UX 분기 명확성을 위해 `verify` 단계에서 `existingUser`까지 같이 주면 더 좋음
- 가능하면 최종적으로는 `verify`에서 기존 회원 여부를 알려주고, `login`은 순수 로그인 처리 역할만 맡는 구조가 더 명확함

## 12. 개인정보 수집·이용 동의 내역 조회

### 필요한 이유
- 설정 화면에 `개인정보 수집·이용 동의 내역` 페이지가 추가됨
- 현재는 프론트 더미 데이터만 사용 중

### 권장 API
- `GET /api/v1/users/me/terms-agreements`

### 응답 예시
```json
{
  "items": [
    {
      "type": "SERVICE_TERMS",
      "title": "서비스 이용 약관",
      "version": "v1",
      "agreed": true,
      "agreedAt": "2026-06-18T12:34:56Z"
    }
  ]
}
```

### 프론트가 필요한 필드
- `type`
- `title`
- `version`
- `agreed`
- `agreedAt`

### 타입 값은 가입 요청과 맞추는 편이 좋음
- `SERVICE_TERMS`
- `PRIVACY_THIRD_PARTY`
- `LOCATION_BASED_SERVICE`
- `AGE_OVER_14`

## 13. 활동 카테고리 메타데이터 조회

### 필요한 이유
- 현재 프론트는 탐색 필터, 생성 플로우, 카드 라벨 표시용 카테고리 값을 상수로 들고 있음
- 카테고리 추가/비활성화/노출순서 변경 시 앱 배포 없이 반영할 수 있게 백엔드 기준으로 맞추고 싶음

### 권장 API
- `GET /api/v1/activity-categories`

### 응답 예시
```json
{
  "items": [
    {
      "code": "TOURIST_ATTRACTION",
      "label": "관광지",
      "displayOrder": 1,
      "active": true,
      "children": [
        {
          "code": "PALACE",
          "label": "궁궐",
          "displayOrder": 1,
          "active": true
        }
      ]
    }
  ]
}
```

### 프론트가 기대하는 최소 필드
- 상위 카테고리
  - `code`
  - `label`
  - `displayOrder`
  - `active`
- 상세 카테고리까지 같이 줄 경우
  - `children[].code`
  - `children[].label`
  - `children[].displayOrder`
  - `children[].active`

### 계약상 중요한 점
- `code`는 표시용 문자열이 아니라 프론트 요청/응답에서 쓰는 계약값으로 유지되어야 함
- 즉, `GET /api/v1/activities` 응답의 `category`, 생성/검색 API에서 받는 `category` 값, 장소 추천의 `category` 값이 모두 같은 코드 체계를 써야 함
- 라벨(`label`)이나 노출순서(`displayOrder`)는 바뀌어도 되지만, `code`를 임의로 바꾸면 프론트 필터/생성 연결이 깨질 수 있음

### 프론트에서 계속 들고갈 값
- `전체(all)` 같은 UI 전용 항목은 서버 응답이 아니라 프론트 고정값으로 유지하는 편이 안전함

### 활용 예정
- 둘러보기 필터 카테고리 목록
- 모임/클래스 생성 시 카테고리 선택
- 필요 시 카드/상세 화면의 카테고리 라벨 매핑 공통화

## 15. 활동 상세의 사용자 상태 응답 보강

### 필요한 이유
- 현재 프론트 활동 상세 화면은 `즐겨찾기 여부`, `내 참여 상태`를 서버에서 못 받아서 로컬 기본값으로만 처리하고 있음
- 이 상태로는 실제 서버 상태와 UI가 어긋날 수 있음

### 요청사항
- 활동 상세 응답 또는 동등한 조회 응답에 아래 중 하나 필요
  - `favorited`
  - `joined`
- 가능하면 boolean 2개보다 상태값 1개가 더 좋음
  - 예: `participationStatus: NONE | PENDING | JOINED`

### 있으면 좋은 확장
- `hostedByMe`
  - 상세 화면에서 호스트 본인일 때 CTA 분기용
- `canJoin`
  - 마감/정원초과/본인호스트 여부를 서버 기준으로 바로 표시 가능

### 관련 API
- `GET /api/v1/activities/{id}`

### 추가 확인 필요
- 현재 프론트 CTA가 `참가하기 / 참가중` 토글 UX라서, 실제 사용자 취소 기능을 제공할 계획이면 취소 API도 필요함
  - 예: `DELETE /api/v1/activities/{id}/participants/me`

## 16. 1:1 채팅방 표시용 제목/상대 정보 보강

### 필요한 이유
- 프론트 채팅 목록은 방 응답의 `title`을 그대로 표시함
- 현재 direct room 제목이 내부 식별 문자열 형태면 실제 UI에 사용자명이 아니라 내부값이 노출될 수 있음

### 요청사항
- direct room 목록 응답에서 최소한 아래 중 하나 필요
  - `title`에 상대방 표시명 세팅
  - 또는 별도 필드 제공
    - `peerDisplayName`
    - `peerProfileImageUrl`

### 관련 API
- `GET /api/v1/chats`

### 프론트 관점 추천
- direct room은 서버가 `title`을 사용자 표시명으로 내려주는 방식이 가장 단순함
- group room과 direct room에서 응답 shape를 최대한 비슷하게 유지하면 연결 비용이 낮음

## 프론트 연결시 중요 포인트 요약
- 차단 목록 / 즐겨찾기 목록 / 참여신청 관리 화면은 모두 `리스트 응답`이 필요함
- 번역 기능은 프론트가 `별도 액션 API`보다 `조회 응답에 원문+번역문 포함` 구조를 더 쉽게 연결 가능
- 기존 회원 분기는 `existingUser`만으로 끝나지 않고 `이전 동네정보`까지 같이 필요함
- 즐겨찾기 목록은 기존 활동 카드 응답 shape를 재사용하면 프론트 수정량이 가장 적음
- 카테고리 메타데이터는 서버에서 내려줄 수 있지만, `code`는 바뀌지 않는 계약값으로 유지되어야 함
- 언어 코드는 검색/생성/응답 전체에서 같은 규칙으로 통일되어야 함
- 활동 상세는 `내 상태`를 서버에서 내려주지 않으면 프론트 로컬 상태와 실제 데이터가 쉽게 어긋남
