## 먼저 공유할 전제
- 아래 항목은 `2026-06-18` 기준 프론트 구현에 맞춰 정리한 백엔드 필요사항이다.
- 프론트는 우선 화면/플로우를 완성했고, 일부는 현재 mock/local state로 동작한다.
- 가능하면 기존 응답 shape를 재사용하고, 꼭 새 API가 필요할 때만 추가해주면 된다.
- 인증 필요한 API는 모두 로그인 세션 기준 `me` 처리가 가능하면 그쪽이 프론트 연결에 유리하다.
- 특히 번역/기존회원 판별처럼 별도 액션 API 없이 기존 조회/인증 응답만 보강해도 되는 항목은 그 방식이 프론트 연결 비용이 가장 낮다.

## 1. 신고하기

### 필요한 이유
- 프론트에는 신고 시트 UI가 이미 있음.
- 현재는 제출 결과를 저장하지 못함.

### 요청사항
- 신고 본문 최대 `500자`
- 이미지 최대 `5장`
- 신고 대상 타입 구분 필요
  - 예: `USER`, `ACTIVITY`, `CHAT_MESSAGE`, `REVIEW`
- 신고자 정보는 서버가 인증 세션으로 식별
- 신고 접수 시간은 서버 저장값 기준 사용
- 접수 시 Discord 알림 전송

### 권장 API
- `POST /api/v1/reports`

### 요청 바디 예시
```json
{
  "targetType": "USER",
  "targetId": "123",
  "reason": "부적절한 메시지를 반복적으로 보냈습니다.",
  "imageUrls": [
    "https://..."
  ]
}
```

### 응답 예시
```json
{
  "id": 1,
  "targetType": "USER",
  "targetId": "123",
  "status": "RECEIVED",
  "createdAt": "2026-06-18T12:34:56Z"
}
```

### 참고
- 업로드는 기존 presigned 업로드 플로우 재사용 가능.
- 프론트는 URL 배열만 있으면 됨.

## 2. 유저 차단하기

### 필요한 이유
- 프론트에 타인 프로필 하단 `유저차단하기`와 설정의 `차단 유저 목록` 화면이 추가됨.

### 요청사항
- 차단 시 기존 친구 관계가 있으면 자동 삭제
- 차단한 상대와 다시 친구 맺기 불가
- 차단한 상대 프로필 상세 조회 불가
- 차단은 자기 자신에게 불가
- 중복 차단 요청은 멱등 처리 권장

### 권장 API
- `POST /api/v1/users/{targetUserId}/block`

### 응답 예시
```json
{
  "targetUserId": 123,
  "blocked": true,
  "friendRemoved": true,
  "blockedAt": "2026-06-18T12:34:56Z"
}
```

### 에러 정책 권장
- 자기 자신 차단 시 `400`
- 이미 차단된 경우 `200` 또는 `409` 중 하나로 통일

### 기존 조회 API에도 필요한 동작
- 최소한 `GET /api/v1/users/{targetUserId}` 호출 시, 내가 차단한 사용자 상세 프로필은 조회되지 않도록 처리 필요
- 응답은 `403` 또는 `404` 중 하나로 통일하고, 프론트 분기용 에러 코드도 함께 주는 편이 좋음
  - 예: `blocked-user`

## 3. 유저 차단해제하기

### 요청사항
- 차단 해제 후 관계는 `친구아닌 일반 유저` 상태로 복귀
- 자동으로 친구 복구되면 안 됨

### 권장 API
- `DELETE /api/v1/users/{targetUserId}/block`

### 응답 예시
```json
{
  "targetUserId": 123,
  "blocked": false
}
```

---

## 4. 차단 유저 목록 보기

### 필요한 이유
- 설정 > 차단 유저 목록 화면 구현 완료
- 각 유저별 이름, 프로필 사진, 거주지역 표시 필요

### 권장 API
- `GET /api/v1/users/me/blocked-users`

### 정렬
- 최신 차단순

### 응답 예시
```json
{
  "items": [
    {
      "userId": 123,
      "displayName": "Lee MinJi 이민지",
      "profileImageUrl": "https://...",
      "activityRegionName": "Living in Seoul · Shindang dong",
      "blockedAt": "2026-06-18T12:34:56Z"
    }
  ]
}
```

### 프론트가 실제로 쓰는 필드
- `userId`
- `displayName`
- `profileImageUrl`
- `activityRegionName` 또는 동등한 거주지역 문자열

### 있으면 좋은 필드
- `blockedAt`
  - 최신 차단순 정렬 확인용
  - 나중에 차단일시 노출이 필요해도 그대로 재사용 가능

## 프론트 연결시 중요 포인트 요약
- 차단 목록 / 즐겨찾기 목록 / 참여신청 관리 화면은 모두 `리스트 응답`이 필요함
- 번역 기능은 프론트가 `별도 액션 API`보다 `조회 응답에 원문+번역문 포함` 구조를 더 쉽게 연결 가능
- 기존 회원 분기는 `existingUser`만으로 끝나지 않고 `이전 동네정보`까지 같이 필요함
- 즐겨찾기 목록은 기존 활동 카드 응답 shape를 재사용하면 프론트 수정량이 가장 적음
- 카테고리 메타데이터는 서버에서 내려줄 수 있지만, `code`는 바뀌지 않는 계약값으로 유지되어야 함
- 언어 코드는 검색/생성/응답 전체에서 같은 규칙으로 통일되어야 함
- 활동 상세는 `내 상태`를 서버에서 내려주지 않으면 프론트 로컬 상태와 실제 데이터가 쉽게 어긋남
